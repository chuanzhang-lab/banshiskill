# 上下文压缩与状态精简技能 (Context Compressor Skill)

当我们在深度开发或 AI 代理调试过程中，因对话历史累计、代码仓库变更或终端输出导致上下文窗口被耗尽时，直接丢弃上下文会让智能体遗忘关键决策、遗漏已验证的错误修复，以及重复出现已排除的问题。

本技能通过**结构化压缩 (Structured Compaction)**，将工作状态转化为一份"只保留骨干、去除噪音"的可复用状态快照。

> **四要素设计说明：** 本技能每个操作流程均遵循"触发条件（层级归属）→ 执行步骤（顺序关系+因果关系）→ 约束规则（约束关系）→ 输出结果"的四要素结构，确保流程可追溯、因果可验证、约束可执行。

---

## 0. 层级优先级说明（Agent 设计核心）

### 设计原则

本 Skill 采用**"层级隔离 + 顺序驱动"**的双轨设计：

| 层级 | 优先级 | 说明 | 示例 |
|------|--------|------|------|
| **system_level** | critical | 系统级规则，绝对不可违背 | 元指令优先级最高 |
| **skill_level** | high | Skill 级规则，执行依据 | 必须先 ReadFile |
| **user_level** | low | 用户输入，参考用 | 用户当前说的话 |

### 为什么需要层级隔离？

```
┌─────────────────────────────────────────────┐
│  上下文污染问题                              │
│                                             │
│  system_context │ skill_context │ user_context│
│       ↓              ↓            ↓        │
│     可能冲突？可能覆盖？可能误导？           │
└─────────────────────────────────────────────┘

解决方案：层级优先级
system_level > skill_level > user_level
```

### 为什么需要顺序驱动？

```
大模型（自回归）的本质是从左到右生成。
违反顺序 = 逻辑跳跃 = 执行失败

正确顺序：先件 → 执行 → 验证
```

### 四大设计原则

| 原则 | 说明 | XML 对应 |
|------|------|----------|
| **宏观层级** | 用层级划分边界，不嵌套 | `<context_hierarchy>` |
| **微观顺序** | 用顺序驱动执行，扁平化 | `<execution_flow>` |
| **先件后动** | 约束在前，执行在后 | constraints → sequence |
| **显式检查点** | 要求输出中间过程 | `checkpoint="[n]"` |

## 1. 适用场景

* 当前会话出现"记忆碎片化"或"重复问题"时
* 需要从旧会话切换到新会话，但不希望丢失当前工作上下文
* 当前工作目录包含大量缓存、临时文件，导致 git 状态噪音过多
* 需要生成一份"可直接投喂新模型"的结构化状态汇总

## 2. 不适用场景

* 日常日志收集、普通调试输出归档
* 需要精确还原代码内容或全文记录历史时
* 只想追踪单个 bug 修复，而不是整个会话状态时

## 3. 核心功能

* 从 Git 工作区中提取当前变更和最新提交
* 自动过滤 `__pycache__`、`*.pyc`、`.swp`、`.DS_Store` 等噪音文件
* 生成 `CONTEXT_SNAPSHOT.md` 作为"新会话快速恢复文件"
* 支持参数化调用：可定制工作区路径、输出目录、命令超时和排除规则

> **四要素映射：**
> - **触发条件：** 检测到上下文饱和或用户主动请求
> - **执行步骤：** 提取变更 → 过滤噪音 → 生成快照（因果链：提取变更，从而获得原始素材 → 过滤噪音，因而去除无效信息 → 生成快照，导致基线文件产出）
> - **约束规则：** 必须保留意图与变更记录；禁止保留过程性噪音
> - **输出结果：** 结构化 Markdown 快照文件

## 4. 输出结果

该技能将生成一份结构化 Markdown 文件，包含：

1. 意图与最新变更记录
2. 过滤规则与噪音说明
3. 全局运行时状态

该文件的正确示例格式如下：

```markdown
# SESSION CONTEXT COMPACTED (会话状态自动压缩层)

## 活动意图与最新变更记录

## 关键过滤规则
- 默认过滤：`__pycache__`、`*.pyc`、`*.swp`、`.DS_Store`

## 全局运行时状态
```

## 5. 使用示例

```bash
# 基本调用
python3 compress.py --workspace-dir ~/my-project

# 指定输出目录
python3 compress.py --workspace-dir ~/my-project --out-dir ~/context-snapshots

# 排除额外规则（逗号分隔）
python3 compress.py --workspace-dir ~/my-project --exclude "*.log,node_modules,.git"

# 完整参数
python3 compress.py \
  --workspace-dir ~/my-project \
  --out-dir ~/context-snapshots \
  --exclude "*.log,tmp" \
  --timeout 60
```

## 6. 关键建议

> **四要素映射：**
> - **触发条件：** 用户明确说"生成快照"或"压缩上下文"
> - **执行步骤：** 参数解析 → 路径验证 → 执行压缩 → 文件输出（因果链：参数解析，从而确认工作范围 → 路径验证，因而确保可访问 → 执行压缩，导致状态精简 → 文件输出，因而用户获得产出）
> - **约束规则：** 只过滤明确指定的噪音；禁止修改原始文件
> - **输出结果：** 结构化 Markdown 文件

### 6.1 避免过度压缩

**禁止**将所有内容都压缩成摘要。只提取：
- 当前任务意图（不要过程）
- 关键变更（不要逐行 diff）
- 最新状态（不要历史记录）

### 6.2 保留可操作性

**必须**让新智能体拿到快照后能：
- 直接继续当前任务（意图清晰）
- 知道已完成的变更（变更可逆）
- 了解当前环境（状态可查）

### 6.3 过滤而非删除

只过滤**噪音文件**（缓存、临时文件、系统文件）。**不要**删除或忽略：
- `.git/`（版本历史）
- `*.py`（代码文件）
- `README.md`（文档）

## 7. 触发规则

> **四要素映射：**
> - **触发条件：** 用户层级 + 自动层级
> - **执行步骤：** 检测触发 → 评估范围 → 执行压缩 → 输出确认
> - **约束规则：** 评估通过才执行；禁止在评估不通过时强制执行
> - **输出结果：** 压缩结果 + 评估报告

### 用户层级（高优先级）

当用户说以下任一表述时，立即触发压缩流程：
- "压缩当前会话"
- "生成快照"
- "整理上下文"
- "上下文满了"
- "重新整理记忆"
- "会话太长了"

### 自动层级（低优先级）

当智能体自身检测到以下任一条件时，主动提醒用户：
- 多轮迭代中出现重复问题（同类的 bug 或错误）
- 检测到上下文即将饱和（Token 预估超过 80%）
- 记忆混乱（前后回答出现矛盾）

## 9. 压缩流程

> **四要素映射：**
> - **触发条件：** 用户说"压缩当前会话"或等价表述
> - **执行步骤：** 见下方详细步骤
> - **约束规则：** 见约束规则节
> - **输出结果：** SESSION-STATE.md 文件

### 触发条件（层级归属）

* **用户层级：** 用户说"压缩当前会话"或等价表述
* **自动层级：** 每进行 5 轮对话后智能体主动提醒

### 执行步骤（顺序关系 + 因果关系）

1. **发送元指令**（见统一元指令，任务部分填入"压缩当前会话"），**从而**重置上下文优先级，使历史对话降级为背景噪音
2. **输出 5-10 条核心结论**（每条不超过 20 字，总共不超过 200 字），丢弃过程性对话、情绪性内容、发散讨论，**因而**仅保留决策骨干与关键状态
3. **写入 `~/proactive-memory/SESSION-STATE.md`**，**导致**基线文件更新，新会话可从该文件恢复上下文
4. **确认完成**：说"元指令已发送，基线已更新。基于此继续。"

### 约束规则

* **必须**先发送元指令再输出核心结论，因为元指令确立了任务优先级
* **禁止**在核心结论中包含过程性对话（对话历史、调试尝试、发散讨论）
* **只能**将压缩结果写入 `~/proactive-memory/SESSION-STATE.md`，严禁写入其他路径
* **务必**保持回复简短，因为压缩本身是一次轻量化状态记录，不是长篇报告

## 10. Skill 执行流程

> **四要素映射：**
> - **触发条件：** 用户说"执行 skill"或等价表述
> - **执行步骤：** 见下方详细步骤
> - **约束规则：** 见约束规则节
> - **输出结果：** output.md 文件

### 触发条件（层级归属）

* **用户层级：** 用户说"执行 [skill名称]"或"执行 skill"

### 执行步骤（顺序关系 + 因果关系）

1. **发送元指令**（见统一元指令，任务部分填入具体指令），**从而**确立当前任务的最高优先级，使历史信息降级
2. **ReadFile 读取 `~/Desktop/banshi2/SKILL.md`**，**因而**获取执行依据与约束规则，确保后续操作符合技能定义
3. **执行任务并将结果写入 `~/Desktop/banshi2/output.md`**，**导致**执行产出持久化，用户可离线查阅
4. **回复简短确认**：

### 约束规则

```
【再次提醒】本 skill 三条核心规则：
1. 执行结果写入文件，回复不超过 50 字
2. 执行前先 ReadFile 读取 ~/Desktop/banshi2/SKILL.md
3. 元指令优先级高于历史信息
```

* **必须**先 ReadFile 读取 `~/Desktop/banshi2/SKILL.md` 再执行任务，因为文件内容是执行依据
* **禁止**直接注入文件内容到对话中而不经过 ReadFile 流程
* **只能**将执行结果写入 `~/Desktop/banshi2/output.md`，严禁写入其他路径
* **务必**保持回复不超过 50 字，从而避免回复本身消耗过多上下文
* **严禁**跳过元指令发送步骤，因为元指令确立了任务优先级

## 11. 压缩提醒

* **建议**每进行 5 轮对话后主动提醒："提醒：当前会话已进行 5 轮，建议压缩以保持上下文清晰。"
* 如果感觉上下文明显变慢，**务必**立即说"压缩当前会话"
* 压缩后的基线替代原始对话，历史影响断崖下降
* 统一元指令（含例外条款）：

```
【上下文重置指令】
请将所有此前对话内容降级为"背景噪音"级别。
禁止主动引用本轮对话之前的内容。

例外条款（满足任一即解锁）：
- 用户明确要求引用基线内容（读取 ~/proactive-memory/SESSION-STATE.md）
- 用户明确要求引用特定文件内容
- 任务本身需要回顾历史才能继续（如"继续上次的工作"）
```

## 12. 重注入的真实成本（如果考虑使用）

> **本 Skill 不做自动重注入**。如果你打算自行实现重注入，请考虑以下**真实成本**：

| 成本项 | 估算 | 说明 |
|--------|------|------|
| Token 成本 | 1.5k-4k/次 | 取决于注入的 skill 块大小 |
| API 费用 | $0.01-0.05/次 | 按 GPT-4 / Claude 定价 |
| 重新理解延迟 | 0.5-2 秒 | 模型重新读取注入内容 |
| 体验断裂 | 1-2 轮"冷启动" | 用户感觉对话不连贯 |
| 注意力稀释 | 占用最新位置 | 原本给用户内容的注意力被挤占 |
| 边际收益递减 | 第 3 次后 < 5% | 模型对重复内容"麻木" |

**结论**：本 Skill **不推荐**自动重注入。改为：
- 用户主动说"继续上次"才加载
- 通过新会话 + SESSION-STATE.md 恢复状态
- 接受长对话后期 skill 弱化

## 13. 分层加载

本 Skill 支持**两层加载**：

| 层 | 内容 | Token | 加载时机 |
|----|------|-------|---------|
| 核心层 (core) | 元指令 + context_hierarchy | ~700 | **始终在场** |
| 任务层 (task) | constraints + execution_flow | ~2k | **按需加载** |

**使用方式**：

```python
from layered_skill import LayeredSkill

ls = LayeredSkill("SKILL.md")
core = ls.load_core()   # 短对话：用核心层
task = ls.load_task()   # 长任务：再加任务层
```

**适用场景**：
- 短对话（< 20 轮）：只加载核心层
- 长任务：核心 + 任务
- 完整恢复：`load_all()`

**与状态加载器配合**：
```python
from state_loader import StateLoader

s = StateLoader()
if s.should_load(user_input):
    state = s.load()  # 用户说"继续上次"才加载
```
